## Problem Statement

Build a faithful Duolingo clone to learn by building. The real goal isn't to compete with Duolingo — it's to understand how Duolingo's behavioral machinery works by implementing it. The learning is in building the streak system, the XP loop, the lesson state machine, and the instant-feedback UI that makes the product habit-forming.

PRD scope: the full Duolingo product spec exists, but the first version is deliberately trimmed to the core learning loop: pick a language → work through a lesson → get instant feedback → earn XP → see a streak.

## What Makes This Cool

The interesting part of Duolingo isn't the flashcard content — it's the behavioral machinery underneath:
- A lesson state machine that feels exactly right (instant feedback, confidence animations, skip nothing)
- An XP/streak system with idempotent event tracking so streaks don't break on weird timezone edge cases
- A completion screen that delivers a dopamine hit (Duolingo's "you're on fire" screen is a real product decision, not decoration)

Codex's "Builder X-Ray Mode" idea — showing the live state machine and event ledger during a lesson — is worth considering as a Phase 2 feature. It turns this from a clone into a portfolio standout.

## Constraints

- Web app (not native mobile). Responsive, mobile-friendly. Not iOS/Android native.
- Content-minimal: a handful of hardcoded Spanish exercises to prove the mechanics. Real course content is a data problem deferred to later.
- Two-person team (Jason + Bath). Design for shared contribution from day one — clear component ownership, no giant files.

## Premises

1. **Web-first**: A Next.js responsive web app covers the "mobile and web" requirement from the PRD. Native app is a separate project.
2. **Content-minimal**: A working lesson with 5-8 hardcoded Spanish exercises proves everything mechanical. Expanding content later is additive, not architectural.
3. **P0 floor, not ceiling**: The PRD lists ~15 P0 requirements. The actual MVP floor is: lesson runner → feedback → XP → streak. Hearts, leaderboards, skill trees, placement tests, and audio are all P1+ for this build.

## Approaches Considered

### Approach A: Minimal Viable Clone
No backend. Hardcoded lessons, local storage for XP/streak. Ship something today, refactor later.
- Effort: S (1-2 days)
- Risk: Low — no infrastructure to set up
- Pros: Something working by end of day; forces clean component boundaries; no auth headaches
- Cons: No cross-device persistence; everything refactors when backend lands; weaker portfolio piece

### Approach B: Full-Stack from Day One *(CHOSEN)*
Next.js App Router + TypeScript + Tailwind + Supabase Auth. Real auth, real persistence, streak that actually works across devices.
- Effort: L (1-2 weeks to solid MVP)
- Risk: Medium — more moving parts, but the right long-term architecture
- Pros: Streaks work across devices; teaches the modern full-stack pattern (server actions, auth, DB); strong portfolio
- Cons: Supabase + auth setup before writing a single quiz question; easy to spend 3 days on auth and never touch the fun stuff

### Approach C: Client-First, Polish the Feel
Nail the lesson interaction and animations until they feel better than Duolingo's, then add a backend.
- Effort: M (3-5 days to demo-able state)
- Risk: Low-Medium
- Pros: The lesson runner is where the fun is; clean component API before any backend coupling
- Cons: Two-phase rework; Vite/CRA instead of Next.js reduces ecosystem

## Recommended Approach

**Approach B — Full-stack from day one**, but with a strict gate: the first milestone is a working hardcoded lesson with XP and streak. Nothing else ships before that loop feels great.

Concrete scope for Milestone 1:
- Next.js App Router scaffold (TypeScript, Tailwind)
- Supabase setup: one `users` table, one `xp_events` table, one `streaks` table
- NextAuth (or Supabase Auth) — email/social login
- One language hardcoded: Spanish
- 5-8 exercises: 3 types — translate English→Spanish, match word pairs, fill in the blank
- Instant feedback UI: green/red with correct answer shown, Duolingo-style bottom drawer
- XP counter: +10 XP per correct answer, accrued client-side, persisted as a single event on completion
- Streak: increments if user completes at least one lesson per calendar day in their stored timezone (see Data Model)

Everything in the PRD outside this list (hearts, skill tree, audio, placement test, leaderboard, speaking exercises) is deferred.

## Data Model

```sql
-- users table (managed by Supabase Auth, extended with)
users (
  id         uuid primary key references auth.users,
  timezone   text not null default 'America/Los_Angeles'  -- IANA zone, e.g. 'America/Los_Angeles', not UTC offsets
)

-- xp_events table (one row per lesson completion)
xp_events (
  id          uuid primary key default gen_random_uuid(),
  user_id     uuid not null references users(id),
  lesson_id   text not null,               -- e.g. 'spanish-lesson-1'
  completed_day date not null,             -- current_date in user's timezone
  xp_earned   integer not null,
  created_at  timestamptz default now(),
  unique (user_id, lesson_id, completed_day)  -- idempotency key: no double-counting
)

-- streaks table
streaks (
  user_id              uuid primary key references users(id),
  current_streak       integer not null default 0,
  last_completed_day   date    -- null if never completed a lesson
)
```

**Streak increment algorithm** (server action, on lesson complete):
1. Compute `today` = current date in the user's stored IANA `timezone` (server-side, never client-supplied)
2. Insert into `xp_events` with `ON CONFLICT DO NOTHING` — this is the idempotency gate
3. If the insert affected 0 rows → return early (duplicate submission, no streak update)
4. Read `last_completed_day` from `streaks` inside the same transaction
5. If `last_completed_day >= today` → no-op (clock skew or same-day)
6. If `last_completed_day == today - 1 day` → `current_streak += 1`, `last_completed_day = today`
7. Else → `current_streak = 1`, `last_completed_day = today` (broken streak or first ever)
8. Upsert `streaks` row (steps 4-7 run inside a serializable transaction to prevent concurrent race)

**Why this order matters**: XP insert comes before streak update. The unique constraint prevents double-XP. The streak update is gated on the insert actually landing, so a double-submit can never double-increment the streak.

**`completed_day` is always computed server-side** from the user's stored timezone. The client never sends a date. Answers are validated client-side (hardcoded TypeScript objects), but the server recomputes the XP total from the exercise count rather than trusting the client-reported number.

**Signup trigger**: Supabase Auth manages `auth.users`. A Postgres trigger on `auth.users` inserts a row into `public.users` with the new user's id and default timezone. Without this trigger, the FK in `xp_events` fails on first lesson completion.

```sql
create function public.handle_new_user()
returns trigger as $$
begin
  insert into public.users (id) values (new.id);
  return new;
end;
$$ language plpgsql security definer;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute procedure public.handle_new_user();
```

**Row Level Security**: All tables have RLS enabled. Server actions use the Supabase service role key (via `SUPABASE_SERVICE_ROLE_KEY` env var, never exposed client-side). Client only calls the anon key for read-only data (current streak display). RLS policies restrict `xp_events` and `streaks` to `auth.uid() = user_id` for the anon role, but writes go through server actions only.

**Note on timezones**: UTC is simple but ships the exact bug the project is designed to understand — a user in UTC-8 who studies at 4:30pm PT crosses a UTC day boundary and loses streak credit. This build stores a `timezone` field and uses it. The tradeoff of UTC-simplicity isn't worth making the core educational mechanic wrong.

**Lesson completion definition**: a user "completes" a lesson when they reach the completion screen. Wrong answers advance the exercise (no re-queuing in Milestone 1 — deferred to Milestone 2 when content is richer). Answers are validated client-side from the hardcoded TypeScript exercise objects — no network round-trip on answer check. The lesson-complete server action fires only on reaching the final screen.

A successful Milestone 1 looks like:
- A logged-in user can complete a full 5-exercise lesson and see a completion screen with their XP earned

- Completing a lesson today and tomorrow builds a 2-day streak
- The lesson feedback UI feels instant (no perceptible lag on answer submission)

## Resolved Architecture Decisions (from /plan-eng-review)

### D1 — Streak transaction atomicity
The streak algorithm runs inside a PostgreSQL function called via `supabase.rpc('complete_lesson', { lesson_id })`. The Supabase JS client cannot do multi-statement transactions via PostgREST; the function provides the serializable boundary.

### D2 — Lesson state machine
Enum + transition function. No XState. `transition()` returns `{ nextState, events: LessonEvent[] }` so the event log is available for X-Ray Mode (Phase 2) without adding a dependency now.

```ts
type LessonEventType = 'ANSWER_SUBMITTED' | 'XP_AWARDED' | 'LESSON_COMPLETED' | 'STREAK_EXTENDED'
type TransitionResult = { nextState: LessonState; events: LessonEventType[] }
function transition(state: LessonState, event: LessonAction): TransitionResult
```

### D3 — Completion callback contract
`onComplete({ xpEarned, newStreak }: CompletionResult)` — fires AFTER the RPC resolves with server-verified data. Completion screen renders only with confirmed server state.

### D4 — Exercise type schema
```ts
// src/content/types.ts — commit this first, before components or content
type Exercise =
  | { type: 'translate'; prompt: string; answer: string; hint?: string }
  | { type: 'match'; pairs: Array<{ left: string; right: string }> }
  | { type: 'fill'; template: string; answer: string; options: string[] }
```
Case-insensitive answer matching for 'translate' and 'fill'. 'match' scoring: all pairs correct = full XP, no partial.

### D5 — Error handling on completion
If RPC throws: completion screen shows "Lesson complete! Progress couldn't be saved. Check your connection." with a Retry button that re-fires the server action.

### D6 — Authorization in server action
```ts
// src/actions/completeLesson.ts
const supabase = createServerSupabaseClient()
const { data: { user }, error } = await supabase.auth.getUser()
if (!user) return { error: 'unauthenticated' }
// user.id comes from the session cookie — never from the client
const { data, error: rpcError } = await supabaseAdmin.rpc('complete_lesson', {
  p_user_id: user.id,   // server-derived, never client-supplied
  p_lesson_id: lessonId
})
```

### D7 — Idempotent retry: return existing row on conflict
```sql
-- In complete_lesson() when INSERT returns 0 rows:
SELECT xp_earned, (SELECT current_streak FROM streaks WHERE user_id = p_user_id)
INTO v_xp, v_streak
FROM xp_events
WHERE user_id = p_user_id AND lesson_id = p_lesson_id AND completed_day = v_today;
RETURN QUERY SELECT v_xp, v_streak;  -- same result as original completion
```
Retry is indistinguishable from first success. User always sees the correct screen.

## Failure Modes

| Codepath | Realistic failure | Test? | Error handling? | User sees? |
|---|---|---|---|---|
| `complete_lesson()` INSERT | Duplicate submission (double-click / retry) | Yes — idempotency unit test | `ON CONFLICT DO NOTHING` + return existing row (D7) | Same completion screen |
| `complete_lesson()` streak update | Concurrent completions race | No explicit test | Serializable transaction (D1) | No visible effect (one wins) |
| `complete_lesson()` timezone lookup | `users.timezone` is NULL (pre-trigger data) | No | Function will error; server action returns error | Error state + Retry (D5) |
| Server action `auth.getUser()` | Session expired mid-lesson | No | Returns `{ error: 'unauthenticated' }` | Error state — **silent gap**: no redirect to login |
| Server action RPC call | Network timeout / Supabase outage | No | Server action returns error | Error state + Retry (D5) |
| `transition()` called with invalid state | Programmer error | Yes — unit tests cover all transitions | TypeScript exhaustiveness check | N/A — compile-time |
| Lesson page: `onComplete` fires | RPC succeeds but UI crashes | No E2E test yet | None — unhandled | Blank/crashed completion screen — **critical gap** |

**Critical gaps:**
1. Session expired mid-lesson → no redirect to login, user sees generic error. Add to Milestone 1 error handling or accept the UX gap explicitly.
2. `onComplete` callback crash (JS error after RPC success) → blank screen with no retry path. Add an error boundary around CompletionScreen.
